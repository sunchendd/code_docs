# Kernel Launch 与图执行机制详解

## 1. Kernel Launch 是什么？

### 1.1 通俗理解："发号施令"

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Kernel Launch = 发号施令                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  场景类比：指挥官（CPU）和士兵（NPU/GPU）                            │
│                                                                      │
│  非 Kernel Launch 方式（全程陪同）：                                  │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐                       │
│  │ CPU:"用 │────►│ NPU:"好 │────►│ CPU:"再 │                       │
│  │ 乘法算 │     │ 的，我 │     │ 算下一 │                       │
│  │ A×B"   │     │ 在算"  │     │ 步"    │                       │
│  └─────────┘     └─────────┘     └─────────┘                       │
│       ↑                                │                           │
│       └────────────────────────────────┘                           │
│                    来回沟通，效率低下                                │
│                                                                      │
│  Kernel Launch 方式（一次性发令）：                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ CPU:"士兵听令！执行以下任务清单："                           │   │
│  │   1. 矩阵乘法 A×B = C                                       │   │
│  │   2. 矩阵乘法 C×D = E                                       │   │
│  │   3. 归一化操作                                             │   │
│  │   ...                                                       │   │
│  │   N. 结果写回内存                                           │   │
│  │ "去吧！完成后通知我！"                                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                       │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ NPU:"收到！我自己按清单执行，不用你每步都提醒"              │   │
│  │    [自主执行所有任务...]                                     │   │
│  │    "报告！全部完成！"                                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Kernel Launch 就是 CPU 把"任务清单"（kernel）一次性发给 NPU        │
│  然后 CPU 就可以去做别的事，NPU 自己执行                           │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 技术定义

```c
// Kernel Launch 的伪代码
void kernel_launch(
    KernelFunction kernel,     // 要执行的函数（GPU/NPU 代码）
    dim3 grid_dim,             // 启动多少个 block
    dim3 block_dim,            // 每个 block 多少个 thread
    void** args,               // 函数参数（输入数据指针等）
    size_t shared_mem,         // 共享内存大小
    Stream stream              // 在哪个流上执行
) {
    // 1. CPU 准备参数（Host 端）
    setup_kernel_arguments(args);
    
    // 2. 将参数拷贝到设备（H2D 传输，如果需要）
    copy_args_to_device(args);
    
    // 3. 构建命令包（Command Packet）
    CommandPacket cmd;
    cmd.kernel_addr = kernel.device_address;  // kernel 在设备上的地址
    cmd.grid = grid_dim;                       // 执行配置
    cmd.block = block_dim;
    cmd.smem = shared_mem;
    cmd.args = device_args;                    // 设备端参数地址
    
    // 4. 将命令写入命令队列（Command Queue）
    // 这是最关键的一步！
    stream->command_queue.push(cmd);
    
    // 5. 通知设备有新命令（可选，取决于驱动实现）
    notify_device(stream);
    
    // 6. CPU 立即返回，不等待执行完成（异步）
    return;  // ← 关键：非阻塞！
}
```

### 1.3 为什么 Kernel Launch 有开销？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Kernel Launch 开销分解                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  每次 Launch 都要做的工作：                                          │
│                                                                      │
│  1. CPU 端准备（~5-10 μs）                                           │
│     ├─ 验证参数合法性                                                │
│     ├─ 计算线程块分配（grid/block）                                  │
│     ├─ 准备命令包（Command Packet）                                  │
│     └─ 写入命令队列                                                  │
│                                                                      │
│  2. 数据传输（如果需要，~10-100 μs）                                  │
│     ├─ 将 kernel 参数从 CPU 内存拷到 NPU 内存                        │
│     └─ 同步等待传输完成                                              │
│                                                                      │
│  3. 设备端调度（~5-15 μs）                                           │
│     ├─ 从命令队列读取命令                                            │
│     ├─ 分配计算资源（Cube Core/Vector Core）                         │
│     ├─ 加载 kernel 代码到指令缓存                                    │
│     └─ 启动执行                                                      │
│                                                                      │
│  单次 Launch 总开销：~20-125 μs（0.02-0.12 ms）                      │
│                                                                      │
│  对于 LLM Decode（单次计算 0.3ms）：                                  │
│  ├─ 10 个 kernel launch：10 × 0.02ms = 0.2ms（占 67% 时间！）        │
│  └─ 1 个 kernel launch： 1 × 0.02ms = 0.02ms（占 6% 时间）           │
│                                                                      │
│  这就是为什么融合算子能减少 launch 次数，大幅提升性能！              │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. 图执行 ≠ FPGA 硬编码

### 2.1 核心区别

```
┌─────────────────────────────────────────────────────────────────────┐
│              NPU Graph vs FPGA 硬编码的本质区别                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  FPGA（现场可编程门阵列）                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                              │   │
│  │   硬件层面重构                                               │   │
│  │   ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                       │   │
│  │   │LUT  │──│LUT  │──│LUT  │──│LUT  │  查找表（逻辑单元）       │   │
│  │   └─────┘  └─────┘  └─────┘  └─────┘                       │   │
│  │      │        │        │        │                          │   │
│  │      ▼        ▼        ▼        ▼                          │   │
│  │   ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                       │   │
│  │   │FF   │  │FF   │  │FF   │  │FF   │  触发器（状态存储）      │   │
│  │   └─────┘  └─────┘  └─────┘  └─────┘                       │   │
│  │                                                              │   │
│  │   特点：                                                     │   │
│  │   • 物理电路重组（烧录 bitstream）                           │   │
│  │   • 真正的"硬件实现"，不是软件                               │   │
│  │   • 功耗低、延迟确定、可并行度极高                           │   │
│  │   • 编程复杂（Verilog/VHDL）                                 │   │
│  │                                                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  NPU Graph（命令序列缓存）                                            │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                              │   │
│  │   软件层面优化                                               │   │
│  │   ┌──────────┐  ┌──────────┐  ┌──────────┐                  │   │
│  │   │Kernel 1  │──│Kernel 2  │──│Kernel 3  │── ...             │   │
│  │   │命令包    │  │命令包    │  │命令包    │                  │   │
│  │   └──────────┘  └──────────┘  └──────────┘                  │   │
│  │        │             │             │                        │   │
│  │        ▼             ▼             ▼                        │   │
│  │   ┌──────────────────────────────────────────┐             │   │
│  │   │        NPU 硬件（固定架构）               │             │   │
│  │   │  ┌─────────┐  ┌─────────┐  ┌─────────┐  │             │   │
│  │   │  │Cube Core│  │Cube Core│  │Cube Core│  │  计算单元    │   │
│  │   │  └─────────┘  └─────────┘  └─────────┘  │             │   │
│  │   │  ┌─────────┐  ┌─────────┐  ┌─────────┐  │             │   │
│  │   │  │Vector   │  │Vector   │  │Vector   │  │  向量单元    │   │
│  │   │  │Core     │  │Core     │  │Core     │  │             │   │
│  │   │  └─────────┘  └─────────┘  └─────────┘  │             │   │
│  │   └──────────────────────────────────────────┘             │   │
│  │                                                              │   │
│  │   特点：                                                     │   │
│  │   • 软件命令序列缓存（不是硬件重构）                         │   │
│  │   • 硬件还是标准 NPU（Ascend/GPU）                          │   │
│  │   • 减少 CPU 参与，但执行的还是标准 kernel                   │   │
│  │   • 易于编程（Python/C++）                                   │   │
│  │                                                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  关键区别：                                                          │
│  ┌──────────────┬─────────────────┬────────────────────────────┐   │
│  │ 维度         │ FPGA            │ NPU Graph                  │   │
│  ├──────────────┼─────────────────┼────────────────────────────┤   │
│  │ 实现层级     │ 硬件电路        │ 软件命令序列               │   │
│  │ 可编程性     │ 烧录 bitstream  │ 缓存 kernel 调用           │   │
│  │ 并行度       │ 极高（真并行）  │ 中等（SIMT/SIMD）          │   │
│  │ 灵活性       │ 极低            │ 高                         │   │
│  │ 延迟确定性   │ 确定（ns 级）   │ 不确定（μs 级）            │   │
│  │ 功耗         │ 低              │ 高                         │   │
│  │ 编程难度     │ 难（硬件描述）  │ 易（软件）                 │   │
│  └──────────────┴─────────────────┴────────────────────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 NPU Graph 到底是什么？

```
┌─────────────────────────────────────────────────────────────────────┐
│                   NPU Graph 的本质 = 命令序列缓存                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  类比：餐厅的点餐系统                                                │
│                                                                      │
│  普通模式（无图化）：                                                 │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐      │
│  │ 顾客:"点│────►│ 服务员  │────►│ 厨师    │────►│ 出餐    │      │
│  │ 宫保鸡丁│     │ 记菜单  │     │ 看菜单  │     │         │      │
│  └─────────┘     └─────────┘     └─────────┘     └─────────┘      │
│       ↑                                                  │          │
│       └──────────────────────────────────────────────────┘          │
│                    每次都要重新点餐、记录                           │
│                                                                      │
│  图化模式（有图化）：                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 第一次："我是老顾客，以后都要这个套餐"                        │   │
│  │        套餐 = [宫保鸡丁 + 米饭 + 汤]                         │   │
│  │        生成套餐卡 #9527                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                       │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 第 N 次："套餐 #9527"                                        │   │
│  │        服务员直接给后厨，不用重新记录                         │   │
│  │        后厨直接按固定流程做，不用看菜单                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  NPU Graph 同理：                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 图化（第一次）：                                              │   │
│  │   CPU:"NPU 听好，以后每次都要执行这段代码："                 │   │
│  │        [Linear -> RoPE -> FlashAttn -> Linear]               │   │
│  │        记录为 Graph #123                                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                       │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 重放（第 N 次）：                                             │   │
│  │   CPU:"执行 Graph #123，输入数据在地址 0x7f8a0000"           │   │
│  │        NPU 直接从命令缓存读取，无需 CPU 参与                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  关键：不是硬件变了，而是"命令如何下发"优化了                         │
│  • 普通：CPU 每次都要准备命令、检查、下发                            │
│  • 图化：CPU 只更新指针，NPU 自己读命令缓存                          │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 能分布式执行吗？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    NPU Graph 的并行执行能力                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  单个 Graph 内部：                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                              │   │
│  │  Graph 是线性序列（不是 DAG）                                  │   │
│  │  Kernel 1 ──► Kernel 2 ──► Kernel 3 ──► ...                   │   │
│  │     │           │           │                                │   │
│  │     ▼           ▼           ▼                                │   │
│  │   顺序执行，不能并行（数据依赖）                               │   │
│  │                                                              │   │
│  │  原因：Transformer 层内是顺序依赖                              │   │
│  │  ├─ Attention 输出是 MLP 的输入                               │   │
│  │  └─ 无法并行执行                                              │   │
│  │                                                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  但可以通过其他方式并行：                                             │
│                                                                      │
│  1. 单个 Kernel 内部并行（SIMT）                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Kernel: FusedAttention                                       │   │
│  │    ├─ Block 0 (threads 0-127)  处理 token 0-31                │   │
│  │    ├─ Block 1 (threads 128-255) 处理 token 32-63              │   │
│  │    ├─ Block 2 ...                                             │   │
│  │    └─ 每个 block 内 128 threads 并行                          │   │
│  │                                                              │   │
│  │  这是 NPU/GPU 的 SIMD/SIMT 并行，不是图级别的并行             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  2. 多卡并行（TP/DP/PP）                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Device 0         Device 1         Device 2                 │   │
│  │  ┌─────────┐      ┌─────────┐      ┌─────────┐             │   │
│  │  │ Graph   │      │ Graph   │      │ Graph   │              │   │
│  │  │ Layer 0-7│◄────►│ Layer 8-15│◄────►│ Layer 16-23│          │   │
│  │  └─────────┘      └─────────┘      └─────────┘             │   │
│  │       │                │                │                   │   │
│  │       └────────────────┴────────────────┘                   │   │
│  │                    AllReduce                                │   │
│  │                                                              │   │
│  │  多卡之间通过通信原语（AllReduce/NCCLOps）并行               │   │
│  │  每个卡上的 Graph 独立执行，结果同步                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  3. Multi-Stream（多流）                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Stream 0:  [Attention] ──► [MLP] ──► [Next Layer]         │   │
│  │       │                                                     │   │
│  │       │  overlap                                            │   │
│  │       ▼                                                     │   │
│  │  Stream 1:  [Communication] ──► [Communication]             │   │
│  │                                                              │   │
│  │  计算和通信重叠，提高利用率                                    │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  总结：                                                              │
│  ├─ Graph 内部：顺序执行（kernel 间有依赖）                         │
│  ├─ Kernel 内部：高度并行（SIMT）                                   │
│  ├─ 多卡之间：通过通信并行（TP/DP）                                 │
│  └─ 与 FPGA 区别：不是硬编码逻辑阵列，是软件命令缓存                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 总结对比

### 3.1 Kernel Launch 总结

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Kernel Launch 一句话                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Kernel Launch = CPU 把"计算任务清单"一次性发给 NPU 的命令           │
│                                                                      │
│  开销来源：                                                          │
│  1. 准备命令包（~5-10 μs）                                           │
│  2. 参数传输（~10-100 μs）                                           │
│  3. 硬件调度（~5-15 μs）                                             │
│                                                                      │
│  图化的收益：把 100 次 launch 变成 1 次，节省 99 次开销               │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 Graph vs FPGA 总结

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Graph ≠ FPGA 硬编码                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Graph = 软件层面的"命令序列缓存"                                     │
│  FPGA = 硬件层面的"电路重构"                                         │
│                                                                      │
│  关键区别：                                                          │
│  ├─ Graph 减少的是"CPU 参与"，不是改变硬件                           │
│  ├─ FPGA 改变的是"硬件电路"，实现真正定制                            │
│  ├─ Graph 易于编程（Python/C++）                                     │
│  └─ FPGA 难以编程（Verilog/VHDL）                                    │
│                                                                      │
│  NPU Graph 不能分布式执行（单个 graph 内是顺序的）                     │
│  但 kernel 内部有 SIMD/SIMT 并行，多卡间有 TP/DP 并行                │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```
