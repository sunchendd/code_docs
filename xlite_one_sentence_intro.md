# Xlite 技术一句话介绍

## 🎯 核心介绍（推荐话术）

**Xlite 是华为昇腾 NPU 上的图化推理引擎，通过"预编译计算图+运行时重放"技术，将 LLM 生成阶段的 Host-Device 交互开销降低至接近零，实现 30-75% 的吞吐量提升；代价是额外占用约 **TensorPool ≈ 0.5-2GB** 显存（与模型规模和并发配置正相关），且仅支持 Llama/Qwen/GLM4 等固定架构。**

---

## 📊 显存占用公式

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Xlite 额外显存占用                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  总额外显存 = TensorPool + RoPE缓存 + HiddenStates预分配             │
│                                                                     │
│  ├─ TensorPool:     由模型内部计算决定（几百MB ~ 几GB）               │
│  │                   主要存储图化后的中间结果和算子状态               │
│  │                   估算公式: ~0.5GB × (hidden_size/4096) × (layers/32)│
│  │                                                             │
│  ├─ RoPE缓存:       max_seq_len × rotary_dim × dtype_size          │
│  │                   例: 32768 × 128 × 2B = 8 MB                    │
│  │                                                             │
│  └─ HiddenStates:   max_num_tokens × hidden_size × dtype_size      │
│                      例: 8192 × 4096 × 2B = 64 MB                   │
│                                                                     │
│  实际额外开销（以 Llama3-8B 为例）:                                   │
│  - TensorPool: ~1-1.5 GB（主要开销）                                 │
│  - RoPE缓存:   ~8 MB                                                │
│  - HiddenStates: ~64 MB                                             │
│  ─────────────────────────                                          │
│  总计: ~1.1-1.6 GB 额外显存                                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 显存对比速查表

| 模型 | 基础显存 | Xlite 额外 | 总显存 | 增幅 |
|------|---------|-----------|--------|------|
| Llama3-8B | ~16 GB | ~1.2 GB | ~17.2 GB | +7.5% |
| Llama3-70B | ~140 GB | ~3-4 GB | ~144 GB | +2.9% |
| Qwen2.5-32B | ~64 GB | ~2-3 GB | ~67 GB | +4.7% |

## 💡 技术特点（三句话版本）

> **Xlite 是华为为昇腾 NPU 开发的 LLM 推理加速器，核心思想是"用空间换时间"：**
> 
> 1. **空间代价**：预分配 Tensor Pool（~1-2GB）和 RoPE 缓存，换取运行时的零动态分配
> 2. **时间收益**：通过图化技术消除 Kernel Launch 开销，Decode 阶段 TPOT 降低 26-43%
> 3. **架构约束**：仅支持 Decoder-only 架构（Llama/Qwen/GLM4），不支持投机解码和流水线并行

---

## ✅ 优势 vs ❌ 劣势

### 核心优势（为什么用）

| 优势 | 数据支撑 | 适用场景 |
|------|---------|---------|
| **吞吐量提升 30-75%** | TPOT 降低 26-43%，QPS 提升显著 | 高并发在线服务 |
| **响应延迟更稳定** | 消除 CPU 调度抖动 | 实时交互应用 |
| **长文本生成快** | Decode 阶段占主导时优势最大 | 代码生成、文章写作 |
| **算子高度融合** | Fused Attention/MLP 减少内存搬移 | 显存带宽受限场景 |

### 主要劣势（什么时候不用）

| 劣势 | 具体表现 | 替代方案 |
|------|---------|---------|
| **额外显存开销** | +1-2GB Tensor Pool | 原生 PyTorch |
| **启动时间增加** | 初始化+图化编译耗时 10-30s | 服务预热 |
| **模型支持有限** | 仅 7 种架构，新模型需适配 | 等待社区支持 |
| **灵活性差** | 不支持动态 batch、变长序列 | decode-only 模式 |
| **调试困难** | 图化后错误信息晦涩 | 开发阶段禁用 xlite |

## 🎤 推荐话术模板

### 场景 1：向技术领导汇报

> "Xlite 是华为昇腾团队开发的图化推理引擎，能在几乎不改动业务代码的情况下，将 LLM 服务吞吐量提升 30-75%，代价是每台机器多占用约 1-2GB 显存。对于我们的高并发场景，投入产出比很高，建议生产环境试用。"

### 场景 2：向同事技术分享

> "Xlite 的核心是'图化'——把模型计算流程预编译成静态图，运行时直接重放，避免 CPU 反复给 NPU 下发指令。这就像把'每次打车'变成'坐专线公交'，固定路线但更快。代价是显存占用增加 1-2GB，且只支持 Llama/Qwen 等主流模型。"

### 场景 3：向客户/业务方介绍

> "Xlite 是华为提供的 LLM 推理加速方案，能让同样的硬件支持更多并发用户，响应速度更快。实测在 8K 输入 1K 输出的场景下，吞吐量可提升 75%，相当于节省了 40% 的硬件成本。"
